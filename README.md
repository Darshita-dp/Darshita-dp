## 💠 About Me

```text
I build business-facing data and software solutions — not just dashboards.
```

- 📊 I work across the full analytics stack: **raw data → SQL models → Python ETL pipelines → Power BI dashboards → React interfaces → automation workflows**. That combination of BI, data engineering, and software thinking is what I bring to a team.
- 🏥 My domain focus is **healthcare operations analytics and digital automation** — revenue cycle, claims and denials, patient access, appointment scheduling, and operational decision support.
- 🔁 I turn analytics into **action**: rules-based prioritization, staff task recommendations, and workflow automation that tell teams *what to do next*, not just what happened.
- 🧠 Where it serves the business problem, I apply **predictive modeling** — such as appointment no-show risk scoring feeding a recommended-action engine.
- 🧾 I care about **clean data, practical systems, and documentation**: governed data models, data dictionaries, validation checks, and dashboards designed around the questions stakeholders actually ask.
- 🎯 I translate **business problems into usable technical products**, and I'm looking for roles where data analysis, BI, software thinking, and automation overlap.

**Open To:** `Data Analyst` · `Data & Software Analyst` · `Healthcare Data Analyst` · `BI Analyst` · `Power BI Developer` · `Business Systems Analyst` · `Process Automation Analyst`

## 🚀 Featured Projects

<details open>
<summary><b>🏥 Healthcare Revenue Cycle Command Center</b></summary>
<br/>

> An end-to-end healthcare revenue cycle analytics and workflow automation platform using SQL, Python ETL, FastAPI, React, and Power BI.

| Aspect | Details |
|:---|:---|
| **Stack** | Python (pandas, numpy) · PostgreSQL · SQL star schema · FastAPI · React + TypeScript · Power BI (DAX) |
| **Data / Scale** | 6,000 claims · 1,200 patients · 110 providers · 9 payers · 7-dimension / 5-fact star schema |
| **Analytics** | 20+ documented SQL queries — denial rates, payer performance, A/R aging (30/60/90), recovery tracking |
| **Automation** | Rules engine generating prioritized follow-up tasks and payer escalation alerts from analytics |
| **Impact** | Explainable 0–100 claim priority score + revenue recovery simulator — tells staff which claim to work first |
| **Repository** | [Healthcare-Revenue-Cycle-Command-Center](https://github.com/Darshita-dp/Healthcare-Revenue-Cycle-Command-Center) |

Healthcare teams lose revenue to claim denials, slow payer payments, and aging A/R — and often discover problems weeks late in static spreadsheets. This platform answers the questions a revenue cycle team asks daily: which claims were denied and why, which payers need escalation, how much revenue is at risk, and **which claims to follow up on first**. A governed star-schema warehouse and Python ETL feed a FastAPI backend, a 6-page React work-queue application, and a 5-page Power BI executive layer with 14+ DAX measures. The transparent claim priority score and revenue recovery simulator turn reporting into prioritized action — every score comes with business-readable reasons, not a black box.

</details>

<details open>
<summary><b>📅 Patient Access & Appointment No-Show Automation Platform</b></summary>
<br/>

> A healthcare scheduling analytics and automation platform that predicts no-show risk, recommends staff actions, and supports waitlist matching.

| Aspect | Details |
|:---|:---|
| **Stack** | Python · Scikit-learn · PostgreSQL · FastAPI · React (recharts) · Power BI · Power Automate (workflow design) |
| **Data / Scale** | Full ETL + feature engineering pipeline · 13 API endpoints · 8 React operational views · 7-page Power BI design |
| **Analytics** | No-show risk scoring (0.72 ROC-AUC, temporal holdout) — top-20% risk band captures 45% of no-shows at ~2× base-rate precision |
| **Automation** | Recommended-action engine (call, targeted SMS, escalation, transport check) + waitlist matching for open slots |
| **Impact** | Prediction → risk band → staff action → waitlist match — cancelled capacity gets refilled instead of expiring |
| **Repository** | [patient-access-noshow-automation-platform](https://github.com/Darshita-dp/patient-access-noshow-automation-platform) |

Roughly 1 in 5 outpatient appointments ends in a no-show, wasting provider capacity and delaying care for waitlisted patients. This platform is a decision-support system for patient access operations, not just a model: every upcoming appointment receives a risk score with a governed threshold policy and model card, a rules-based engine converts each score into a concrete staff step with a plain-language reason, and a waitlist matching engine ranks replacement candidates for every open or released slot. A React scheduling app and FastAPI backend serve the daily workflow, while a Power BI executive design and Power Automate/SharePoint workflow spec cover management visibility and process automation.

</details>
